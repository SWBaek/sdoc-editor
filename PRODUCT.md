# Structured Doc Editor — Product Vision

Structured Doc Editor는 기술 문서의 구조를 보존하면서 WYSIWYG로 편집하고, Git으로 변경을 관리할 수 있게 하는 오픈 소스 문서 편집기입니다.

이 문서는 제품의 방향과 범위를 설명합니다. 현재 구현의 모듈 경계와 데이터 흐름은 [아키텍처 문서](docs/architecture.md), 저장 형식은 [`sdoc.schema.json`](sdoc.schema.json), 기여 절차는 [CONTRIBUTING.md](CONTRIBUTING.md)를 기준으로 합니다.

## 해결하려는 문제

기술 문서는 제목 계층, 표와 그림, 수식, 다이어그램, 캡션, 교차 참조처럼 Markdown만으로 편집하기 불편한 구조를 자주 포함합니다. 반대로 일반적인 바이너리 문서 형식은 Git에서 변경 내용을 검토하고 자동화 도구로 처리하기 어렵습니다.

Structured Doc Editor는 다음 사이의 간극을 줄이는 것을 목표로 합니다.

- 작성자는 문서 화면에서 구조와 결과를 확인하며 편집합니다.
- 리뷰어는 사람이 읽을 수 있는 JSON과 안정적인 ID를 Git으로 추적합니다.
- 도구는 명시적인 스키마를 기준으로 문서를 검증하고 변환합니다.
- 같은 문서는 VS Code와 Windows 데스크톱 앱에서 동일한 의미로 저장됩니다.

## 대상 사용자

- 설계서, 기술 보고서, 의사 결정 기록, 연구 노트를 작성하는 개발자와 엔지니어
- 문서를 코드와 같은 저장소에서 버전 관리하고 리뷰하려는 팀
- 하나의 원본에서 HTML, PDF, Markdown, AsciiDoc, 슬라이드를 만들려는 작성자
- VS Code를 사용하지 않아도 구조화 문서를 편집해야 하는 Windows 사용자

## 제품 원칙

### 문서가 제품보다 오래 살아야 합니다

`.sdoc`은 버전이 명시된 공개 JSON 계약입니다. 저장 파일에는 portable 상대 경로와 문서 의미만 남기고, 호스트 전용 URL이나 런타임 상태는 저장하지 않습니다. 손상되었거나 지원하지 않는 미래 버전은 원본을 덮어쓰지 않습니다.

### 두 호스트는 하나의 문서 의미를 공유합니다

VS Code 확장과 Tauri 앱은 문서 모델, 설정 해석, 번호 매기기, 교차 참조, 변환기를 `shared/`에서 공유합니다. 호스트별 파일 시스템과 창 동작은 typed adapter 뒤에 둡니다.

### 작성 경험과 검토 가능성을 함께 개선합니다

에디터의 편리함 때문에 저장 형식의 안정성을 희생하지 않습니다. 제목·그림·표·수식의 지속 ID, 예측 가능한 정규화, 스키마 검증을 통해 작은 편집이 불필요한 대규모 diff로 번지지 않게 합니다.

### 내보내기는 같은 문서 의미의 표현입니다

에디터 미리보기와 각 export 형식은 동일한 번호·캡션·교차 참조 규칙을 사용해야 합니다. 형식별 제약이 있을 때는 조용히 손실시키기보다 진단하거나 문서화된 fallback을 사용합니다.

### 로컬 파일 접근은 최소 권한입니다

문서, 이미지, Draw.io, `.sdocbook` 경로는 host 경계에서 검증합니다. 문서 또는 workspace 밖으로의 traversal과 symlink 탈출을 허용하지 않으며, 외부 입력은 검증 전까지 신뢰하지 않습니다.

## 현재 제품 범위

| 영역 | 범위 |
|---|---|
| 편집 | H1–H6, 목록, 표, 이미지, 코드, KaTeX 수식, Mermaid, Draw.io, 인용문, 콜아웃 |
| 문서 구조 | 자동 번호, 안정적인 ID, 교차 참조, 목차, 그림·표 목록, 문서별 설정 |
| 파일 | `.sdoc`, 레거시 `.tiptap.json`, 여러 문서를 묶는 `.sdocbook` |
| 변환 | Markdown·HTML 가져오기, HTML·PDF·Markdown·AsciiDoc·reveal.js 내보내기 |
| 호스트 | VS Code 확장, Windows Tauri 데스크톱 앱 |
| 안전성 | 스키마 검증, revision 기반 저장, 원자적 desktop 저장, asset 경로 격리 |

정확한 지원 여부와 설치 방법은 [README.md](README.md)를 따릅니다.

## 핵심 사용자 흐름

### 단일 문서 작성

1. 새 `.sdoc`을 만들거나 기존 문서를 엽니다.
2. 구조화 편집기에서 제목, 콘텐츠, 캡션과 참조를 작성합니다.
3. 저장 시 portable JSON으로 정규화하고 검증합니다.
4. 현재 문서 내용과 동일한 설정으로 필요한 형식에 내보냅니다.

### 여러 문서를 한 권으로 관리

1. `.sdocbook` manifest에 장의 순서를 정의합니다.
2. 각 장을 독립된 `.sdoc`으로 편집하고 리뷰합니다.
3. preview에서 누락된 문서, 중복 ID, 깨진 참조와 경로 문제를 확인합니다.
4. 검증된 합성 결과를 하나의 산출물로 내보냅니다.

### Git 기반 리뷰

1. `.sdoc`과 로컬 asset을 같은 저장소에 둡니다.
2. 변경된 JSON 구조와 콘텐츠를 diff로 검토합니다.
3. 자동 검사에서 스키마, 참조, 경로와 변환 회귀를 확인합니다.

JSON의 줄 단위 diff는 구조가 복잡할수록 한계가 있습니다. 장기적으로 semantic diff나 전용 리뷰 화면을 추가할 수 있지만, 원본 형식과 일반 Git 도구의 호환성을 포기하지 않는 방향을 우선합니다.

## 비목표

현재 다음 항목은 제품의 기본 책임으로 보지 않습니다.

- 실시간 다중 사용자 공동 편집이나 서버 기반 동기화
- 범용 워드프로세서의 모든 페이지 레이아웃 기능 재현
- 독점 문서 형식의 완전한 round-trip 호환
- 문서 내용을 외부 AI 서비스로 자동 전송하거나 특정 AI 프로토콜에 결합
- 검증되지 않은 플러그인이 로컬 파일에 광범위하게 접근하는 확장 모델

새 기능은 문서 계약의 portability, 두 호스트의 일관성, 데이터 손실 위험을 먼저 평가합니다.

## 향후 방향

우선순위는 GitHub Issues와 실제 사용 사례를 기준으로 결정합니다. 특히 다음 영역을 점진적으로 개선할 수 있습니다.

- 반복 가능한 기술 문서 template과 조직별 profile
- 구조를 이해하는 semantic diff 및 리뷰 보조
- 대형 문서와 `.sdocbook`의 부분 갱신 성능
- 접근성, 키보드 조작, 진단 UX
- import/export fidelity와 자동 호환성 fixture
- 다른 데스크톱 운영체제 지원 가능성 검증

아이디어를 제안할 때는 해결하려는 사용자 문제, 저장 형식 영향, 두 호스트 동작, migration과 테스트 계획을 함께 설명해 주세요.

## 성공 기준

Structured Doc Editor가 다음 조건을 꾸준히 만족할 때 제품 방향이 올바르다고 봅니다.

- 사용자가 편집·저장·재열기 과정에서 콘텐츠나 참조를 잃지 않습니다.
- 같은 fixture가 두 호스트에서 의미적으로 동일하게 저장되고 표시됩니다.
- 내보낸 형식의 번호와 참조가 에디터와 일치합니다.
- 잘못된 문서와 경로가 기존 파일이나 workspace 밖의 파일을 손상시키지 않습니다.
- 새 기여자가 문서와 자동 검사만으로 변경 범위와 품질 기준을 이해할 수 있습니다.
